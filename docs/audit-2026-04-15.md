### Enrichment match rate ŌĆö 93.60%

Confirmed via `public/data/*.json`: 6445 cards with `displayName` out of 6886 total.

**All-or-nothing per set.** Every enriched set is 100%, every unenriched is 0%. The 10 zero-match sets:

| Slug | Cards | Category |
|---|---|---|
| `organized-play-promos` | 134 | OP |
| `sector-and-regional-promos-season-1` | 129 | regional |
| `judge-promos` | 55 | judge |
| `secrets-of-power-weekly-play-promos` | 40 | weekly |
| `shadows-of-the-galaxy-weekly-play-promos` | 20 | weekly |
| `spark-of-rebellion-weekly-play-promos` | 20 | weekly |
| `twilight-of-the-republic-weekly-play-promos` | 20 | weekly |
| `ashes-of-the-empire` | 13 | ? |

Root cause at `scripts/enrich-cards.ts:31-44` (SET_CODE_OVERRIDES) + `enrichment.ts:114-117` ŌĆö when `slugToCode[card.set]` is undefined, we synthesize a fake baseCardId and skip the swuapi join.

**The fix is not uniform.** Some of these (weekly-play promos for SOR / SHD / TWI / SEC) are probably resolvable by adding a set-code override. Others (OPP, SRP, JP) may not exist in swuapi's set taxonomy at all, in which case the zero-match is correct ŌĆö these cards simply live outside that data source.

**Recommendation:** spike ~15 min fetching `https://api.swu-db.com/sets` (or whatever swuapi's list endpoint is) and diff against our `SETS` table; pick up every real mapping in one go, document the residue that truly has no upstream match, and add a fail-loud check at build time when a known-mapped slug drops to 0% enrichment (regression guard).

### SRP parenthetical labels are not variants

`sector-and-regional-promos-season-1.json` (129 rows):

- 33 Standard
- 20 tournament-proof strings (`Champion` / `Finalist` / `Top 4` / `Top 8`, 5 each)
- 77 rows where the parenthetical holds a *collector number* (`"(38)"`, `"(42)"`, `"(131)"`, etc.) instead of a variant label

These all flow through `extractVariantLabel` (`src/variants.ts:102-106`) which just takes the last parenthetical. The pickers then render them as "unknown variants" with the gray fallback pill.

**Why it happens:** the fetch script in `scripts/fetch-prices.ts` emits a variant field by pattern-matching TCGPlayer's productName parenthetical. SRP cards don't have printing variants in the Gold/Showcase sense ŌĆö they're prize cards at regional tournaments ŌĆö but TCGPlayer encodes per-event distinctions in the parenthetical.

**Options (not mutually exclusive):**

1. **Filter SRP at fetch time** to drop rows whose parenthetical is purely numeric. ~10 min; removes 77 noise rows.
2. **Filter the tournament-proof labels** too ŌĆö they're not first-class variants. ~10 min.
3. **Document as "regional-exclusive, unenriched"** in the roadmap and leave be.

The user's tradeoff: SRP cards *do* have real prices, so filtering them eliminates real inventory from the picker. Labeling them cleanly (option 1+2 but *keeping* them) may be better than dropping them. This deserves a user call.

### ­¤ö┤ Duplicate productId rows in the dataset (real bug)

`scripts/fetch-prices.ts` is emitting the same productId twice in 8 sets. 34 duplicates total:

| Set | Duplicate rows |
|---|---|
| `secrets-of-power` | 13 |
| `jump-to-lightspeed` | 9 |
| `legends-of-the-force` | 7 |
| `a-lawless-time` | 2 |
| `organized-play-promos` | 1 |
| `spark-of-rebellion` | 1 |
| `twilight-of-the-republic` | 1 |

Confirmed truly identical: I pulled both rows for `654109` (Luthen Rael, SEC) ŌĆö every field is byte-equal. Not a variant collision, not a mid-print re-tag. It's a fetch-loop bug ŌĆö probably a duplicated group or pagination overlap.

**Impact:**
- `App.tsx:147-161` / `ListsDrawer.tsx:52-66` build `byProductId` Maps that silently last-writer-win, so the trade/wants resolution still works.
- But `browseAllGroups` (`useCardSearch.ts:29-61`) doesn't dedupe by productId, so the picker grids and `SearchResults` will render the same tile twice for these 34 cards. Visible in the available picker and trade-search overlays.
- `groupCards` (`variants.ts:142-154`) groups by baseName, so duplicates end up in the same group's `variants` array and render as back-to-back tiles.

**Recommendation:** fix in `scripts/fetch-prices.ts` (dedupe by productId before emit) *and* add an `assertUnique` check in the build so this regresses loudly if it reappears. Cheap; ~30 min.

### Token / leader collisions

**Resolved.** Sampled SHD #001 = "Gar Saxon - Viceroy of Mandalore" (Leader). `enrichment.ts:51-81` correctly suppresses tokens when a real card type is present at the same canonical id. No current dataset has `cardType === 'Token Unit'` or `'Token Upgrade'` leaking through ŌĆö `enrich-cards.ts` drops them at the cleaning step.

Worth keeping the build-time assertion so a future swuapi change can't silently regress this.

### Error paths

**Contrary to the probe, `App.tsx:303-320` already renders errors** with a Retry button. What's not great:
- **Manifest failures are silently swallowed** (`usePriceData.ts:28`) ŌĆö the `priceTimestamp` just stays null and the footer shows nothing. If the manifest 404s in production, the user sees no price-freshness indicator and has no idea. Minor UX gap.
- **Discovered-set errors render without a working retry.** `App.tsx:309-314` calls `SETS.find(...)` to build the retry payload, so a set loaded via `manifest.json` discovery (`usePriceData.ts:63-73`) that isn't in our local `SETS` table will show the error but `Retry` silently no-ops. Edge case; unlikely in practice since `SETS` covers everything we ship.

### `normalizeCardNumber` edge cases

`enrichment.ts:35-40` + `src/enrichment.test.ts`. Current regex handles plain `"5"`, padded `"005"`, and `"224/264"`. Sampled real data: leader cards are `"001/262"`, dual-faced cards appear as `"290 // T04/T04"` which `split('/')[0].trim()` correctly resolves to `"290"`. No shape in the current dataset breaks the function. Comprehensive fixture coverage in the test file; no gap.

### `extractVariantLabel` / `extractBaseName`

Regex-based, `variants.ts:102-110`. The only problematic shapes in real data are the SRP parenthetical-as-number and tournament-proof labels (see above). Swu-api enriched rows bypass this by using `displayName` directly (`variants.ts:138-140`), so the damage is limited to unenriched promo sets ŌĆö same footprint as the enrichment gap.

### Cron health

`.github/workflows/refresh-prices.yml` fires every 2h, runs `curl -s -X POST "${{ secrets.VERCEL_DEPLOY_HOOK }}?buildCache=false"`. The script:
- Uses `-s` (silent) and doesn't check HTTP status, so a dead or rotated deploy hook would fail silently ŌĆö workflow stays green, prices go stale.
- Last real-data timestamp is 2026-04-14 (one day old) ŌĆö plausibly working, but the signal is weak.

**Recommendation:** add the tiny HTTP-status check (capture with `-w "%{http_code}"` and exit non-zero on !=200). 10 min. Parked roadmap item "Decouple price refresh from deploys" (`ROADMAP.md:136-149`) is a better long-term fix but out of scope for this pass.

### Foil-toggle gap for SOR / SHD / TWI

Already parked in `ROADMAP.md:161-171` with a concrete implementation sketch. Assessment stands ŌĆö half-day of work, only matters if someone's asking about early-set foil prices, no user has. Defer.

## 2. Type safety / invariants

Mostly fine. `CardVariant` interface (`src/types/index.ts:10-30`) has every enrichment field optional because legacy cached data (pre-enrichment) needed to validate. That's correct as a contract with disk state, but it leaks everywhere that consumes post-enrichment data ŌĆö every `card.displayName ?? card.name` fallback is extra work for a case that in practice doesn't happen for cards in our index.

**Could introduce** an `EnrichedCardVariant` type and narrow at the `usePriceData` boundary so downstream code can stop writing `card.displayName ?? ŌĆ”`. Moderate-effort refactor across many files; low value because the fallback is cheap and the current code is self-documenting. **Defer.**

`VariantRestriction` discriminated union is used correctly everywhere I checked ŌĆö consumers exhaust both `mode: 'any'` and `mode: 'restricted'` branches (listMatching, ListRows.RestrictionEditor, urlCodec, useWants.restrictionKey). Nothing to change.

Zod ŌåÆ TS chain in `src/persistence/schemas.ts`: types use `z.infer` off the schemas (`schemas.ts:52-54`). No drift. Good shape.

`CardVariant.variant` is typed as `string` where most callers expect a canonical label. This IS a real type-safety gap ŌĆö `variantBadgeColor`, `variantDisplayLabel` etc. all take `string`, and unknown values fall back silently (gray pill). Could narrow to `CanonicalVariant | (string & { __brand: 'unknown' })` but every gain has a corresponding fallback-deletion cost. **Defer** unless we add the Gold/Rose Gold ŌåÆ CANONICAL_VARIANTS work, in which case fold it together.

## 3. Component DRY opportunities

### TradeSide.tsx ŌĆö 838 lines

Composed of:
1. Helper functions (price formatting, `CardThumb`, color maps, `thumbSize`) ŌĆö lines 60-170
2. Outer `TradeSide` function ŌĆö lines 171-769, the bulk
3. `SourceChip` (~30 LOC), `AddCardsTile` (~35 LOC)

Internal structure is:
- Search overlay (fullscreen when active) ŌĆö closing / dismissing logic, filter bar, grid
- Panel shell ŌĆö header with collapse chevron, card-list render, sticky "Add Card" footer, `AddCardsTile` empty-state CTA
- Trade-row render inside the card list (lines 577-738, ~160 LOC of JSX + style logic per row)

**Extraction candidates** (ordered by safety):
1. **`TradeRow`** ŌĆö the 160-line row render. Pure visual component; its inputs are `card`, `qty`, `percentage`, `priceMode`, `tSize`, handlers. Safe, ~30% of the file, measurable size win.
2. **`TradeSearchOverlay`** ŌĆö the full-screen search surface (lines 376-508). Owns the filter bar + source chips + grid + Done button. Its props would be the source/filter state + handlers. This is the second-biggest chunk.
3. **Helpers (color maps, `thumbSize`, `CardThumb`)** ŌĆö move to a sibling file; they're self-contained.

These three extractions would halve the file. I'd recommend one PR each so regressions are isolated.

**Be careful:**
- The color-reservation invariant is critical. Saber-bar, qtyBtn, etc. stay keyed off `accentColor`. Don't substitute generic system colors.
- The search-overlay animation (opacity/translate, `showOverlay` derived from `searchFocused || hasQuery`) is tangled with the state. An extraction must preserve the existing animation semantics.

### Row renderers ŌĆö three similar but non-identical

- **Trade row** (`TradeSide.tsx:577-738`): thumbnail + basename + variant pill + (missing-price icon) + spread badge + qty stepper + kebab + line total.
- **WantsRow / AvailableRow** (`ListRows.tsx:82-138, 279-316`): slightly fancier shell with a restriction editor (wants only); shares thumbnail + title + variant pill + qty stepper + remove.
- **ListView.ListRow** (`ListView.tsx:454-515`): compact variant for the shared-list landing; thumbnail + title + set-code pill + variant/restriction pill + qty + price.

**Assessment:** three uses, different data needs (trade row has spread, list-view row has restriction summary but no stepper, ListRows have restriction *editor* but no spread). This is genuinely just below the "5 rows makes the abstraction obviously worth it" line from CONTINUATION.md.

**Lighter option:** extract `<VariantBadge />` and `<CardThumb />` and let the three rows compose. That collapses the most-repeated pieces without forcing a "one row to rule them all" shape that doesn't exist yet.

Counts of variant-pill markup duplication I can see (by grepping for `variantBadgeColor`):
- `src/components/TradeSide.tsx` (trade row)
- `src/components/ListRows.tsx` (wants + available)
- `src/components/ListCardPicker.tsx` (PickerTile)
- `src/components/ListView.tsx` (ListRow + ListView chip filter)
- `src/components/SelectionFilterBar.tsx` (variant chips)

Six call sites. `<VariantBadge variant="Hyperspace Foil" />` that encapsulates color + display-label + pill shell would be a clean win.

### Filter bars ŌĆö SelectionFilterBar vs ListView inline

`ListView.tsx:255-343` reimplements the CollapsibleChipFilter + Variant + Set chips inline because its filter state is ephemeral (local `useState`) while `SelectionFilterBar` takes the persisted `SelectionFilters` API. They're visually identical.

**Option A:** make `SelectionFilterBar` accept either a `SelectionFilters` API or a `{ selectedVariants, setSelectedVariants, selectedSets, setSelectedSets }` plain-state shape. Small API widening.

**Option B:** keep the ListView inline version but dedupe by extracting `<VariantChipGroup />` and `<SetChipGroup />` that don't care about persistence.

I lean toward B ŌĆö it's more composable and matches the component-DRY direction.

### Popover menu-item shapes

`SharePopoverButton` in `ListsDrawer.tsx:500-519` and the `KebabMenu` item shape in `TradeSide.tsx:687-713` use slightly different chrome for what's visually "menu item with icon + label". Worth reconciling IF we're already touching this area, not a standalone priority.
