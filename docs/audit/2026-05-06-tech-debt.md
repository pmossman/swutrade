## Phase 1.5 — Meta-pass (root-cause groupings + prevention levers)

### Recurring root causes

The 70+ findings cluster into ~8 root causes. Fixing the cause once eliminates the category.

#### MP-1 · No "small primitive" component library
The biggest pattern across A/C/D: components that should share a base instead reach for fresh JSX every time. `LoadingState`, `ErrorState`, `EmptyState` exist but are unevenly adopted (F-A2 through F-A6, F-C1 through F-C7); button-disabled chrome drifts (F-A11, F-D5); `<ConfirmButton>` doesn't exist so two-tap-confirm is duplicated (F-A1) and `window.confirm` is the lazy fallback in 4+ places (F-D1 through F-D4).

**Lever:** ship `src/components/ui/` with `LoadingState`, `ErrorState`, `EmptyState`, `SuccessState`, `ConfirmButton`, `ConfirmDialog`, `Pill`, `Banner`. The first three exist; the rest don't. **Each canonical primitive ships with one ESLint rule (or codebase comment) saying "don't roll your own."** Consolidating the duplicates into adoption commits forces the rest of the app to converge.

#### MP-2 · No fetch convention enforcement
`apiClient.ts` exists with the right shape (`ActionResult<T>`, status→reason mapping) but several components reach past it for direct `fetch()` (F-B6 through F-B9, F-E7) — and no rule prevents the next person from doing it again. Wire-format drift goes unnoticed because schema validation is opt-in (F-E1, F-F8).

**Lever:** (a) make `schema:` required on apiGet/apiPost/etc. for new call sites; (b) add an ESLint custom rule banning `fetch(` outside `src/services/apiClient.ts` and `src/services/livePrices.ts`; (c) write the missing `apiPatch`/`apiDelete` if any are missing; (d) Zod-schema the top-N hot paths incrementally — three this week, three next.

#### MP-3 · Race conditions hidden behind a single mutex pattern that doesn't scale
`mutationInFlightRef` works for poll-vs-mutation but fails for mutation-vs-mutation (F-B1, F-B2, F-B3). The wiki teaches the pattern as if it's complete; in practice there are at least three places (saveCards, confirm/cancel, suggest) where a fast double-click can produce stale state.

**Lever:** generalise to a per-session or per-mutation gen-counter that tags each request with a generation and rejects stale completions. `useGuildMemberships` already does this; promote it to a shared `useMutation` hook with the gen-counter built in. Document in `architecture.md`.

#### MP-4 · Two parallel sources of truth for several types
`CardVariant` exists twice (F-M1); `WantsItem`/`AvailableItem` API shapes don't validate against Zod (F-M2); search redefines its own card types (F-M3); the `communicationPref` enum is duplicated between schema and registry (F-M5). Each duplicate is one rename away from a runtime crash.

**Lever:** establish "schema is source of truth, types flow out via $inferSelect/$inferInsert + Zod schemas" — and ban hand-rolled parallel types via a comment-policy enforced on PR review (or a custom lint). The card-types in `src/types/index.ts` and `src/persistence/schemas.ts` should consolidate.

#### MP-5 · Confirmation UX is three-headed
`window.confirm` (F-D1-D4), two-tap-confirm (F-A1, ListRows + ClearAllButton), and Radix Dialog (HandlePickerDialog) all coexist for the same job. Beta users will eventually report different feel between equivalent destructive actions — and we'll have to retrofit again.

**Lever:** pick one canonical pattern per scope: **two-tap-confirm for in-row destructive** (cheap, low-friction); **`<ConfirmDialog>` for cross-cutting destructive** (more friction for higher stakes). `window.confirm` banned outright. Document in a new `docs/wiki/k-ui-primitives.md`.

#### MP-6 · Dead-code cruft from each rename
Phase C did a clean job overall but left ~15 stale comments / wiki notes / migration-cruft items (F-K1-K7, F-N1-N5). This is the same shape as the pref-registry residue from earlier consolidations. Each rename leaves a fingerprint we never re-grep.

**Lever:** add a one-liner to `feedback_workflow` memory: "after a rename or deletion slice, grep the codebase for the old name + its docs and clean refs in the same commit." Or better, ship a cleanup script (`scripts/cleanup-rename-residue.mjs <old-name>`) that does the find + echoes hits.

#### MP-7 · `localStorage` access scattered
F-B5, F-B14, F-B15: three or four parallel localStorage paths with handwritten try/catch and TTL handling — exactly the seams where the Safari-private-mode bugs hide. There's a `src/persistence/index.ts` already; not everyone uses it.

**Lever:** make `usePersistedState<T>(key, initial, schema?)` the canonical path; `localStorage` outside that file becomes a lint warning. Migrate the four offenders.

#### MP-8 · Visual-token discipline is convention-only
F-I1 (Discord brand color in two files), F-I2 (shadow RGBA hardcoded), F-I3 (inline style), F-I4 (rounded scale undocumented) — the palette memory says "pick from CSS-header table, don't inline hex" but nothing enforces it.

**Lever:** ship `eslint-plugin-tailwindcss` with our token allowlist; ban arbitrary `bg-[#…]` / `text-[#…]` / `shadow-[…]` outside `index.css` + `tailwind.config.js`. Fix the half-dozen offenders.

### Library opportunities (free/open-source only)

- **`zod` (already a dep)** — the right tool for runtime schemas at the API/storage boundary; needs adoption push, not a new dep.
- **`eslint-plugin-react-hooks` (already a dep)** — already enforces exhaustive-deps; verify other rules in the plugin (set-state-in-effect, refs-in-render) aren't disabled.
- **`eslint-plugin-tailwindcss`** — would catch most of MP-8 automatically. Free, MIT, low-risk dep.
- **`@tanstack/react-query`** — would replace ~half the bespoke hook+cache+mutex+poll pattern with a battle-tested implementation. **🔴 Architectural** — bigger discussion, not autonomous; flagged for morning.
- **`vaul` or `@radix-ui/react-dialog`** — already using Radix Dialog in some places; if we're consolidating overlays (MP-5 / F-A7-A9), going all-in on Radix's full primitive set is the path. Already a dep; just adoption.
- **No new deps recommended for tonight's work.** Every prevention lever above can be done with the current dep tree + ESLint rules.

### Bigger architectural questions (🔴, for morning discussion)

- **Q1 — Adopt @tanstack/react-query?** It would obsolete the hand-rolled cache/mutex pattern across `useSession`, `useMyTrades`, `useActiveSessions`, `useTradeDetail`, etc. Big migration, big payoff. Risk: a year-long refactor commitment.
- **Q2 — Migrate fully off ad-hoc overlay paradigms onto Radix?** TradeSearchOverlay, ListsDrawer, FilterPopover all reimplement Dialog/Drawer/Popover. Migration sized at ~5-10 commits but invasive UX-wise.
- **Q3 — Server response shape standardisation.** Some endpoints return `{error}`, some `{error, detail}`, some plain status. Pick one (e.g., RFC 7807 problem-details, or simple `{ok, data?, error?}`) and migrate. Touches every endpoint + apiClient.
