# UX review — 2026-04-17

Four parallel agent passes (core trade flow, social+onboarding,
proposal lifecycle, mobile+a11y+consistency), plus spot-verification
in the running codebase. Findings consolidated and triaged by
user-facing impact.

**Verified against the code:**

- Agent 4's claim "no global `:focus-visible` styling" — confirmed.
  `src/index.css` has one `:focus-within` rule for hover-reveal only.
- Agent 4's claim "header pattern duplicated" — confirmed. The
  Logo + SWUTrade wordmark + BetaBadge chrome appears identically
  across 7 files: `App.tsx`, `ProfileView`, `CommunityView`,
  `SettingsView`, `TradesHistoryView`, `TradeDetailView`, `ListView`.
- Agent 1's claim "qty stepper buttons are ~24px" — confirmed.
  `ListRows.tsx:14,23,37,121,196` all use `w-6 h-6`. WCAG 2.5.5
  wants 44×44.
- Agent 1's claim "wants/blue and available/emerald mapping is
  inconsistent" — FALSE in this app. Confirmed Offering=emerald,
  Receiving=blue in App.tsx:522,554. The naming/color map is
  actually consistent; what's inconsistent is the CONCEPTUAL link
  between "wants" and "Receiving" (see F1 below).

## Top-line read

Three themes dominate:

1. **First-impression clarity is weak.** The balancer's core
   purpose isn't obvious on empty state. Sign-in value prop reads
   generically. Deep-linked flows (Counter from Discord) drop users
   cold into a composer with no "here's what this is" moment.
2. **Design system primitives aren't extracted.** The same header,
   button, status badge, and chip patterns are re-implemented per
   view. Every new feature re-invents these, and they visually drift.
3. **Accessibility + mobile basics miss.** No focus-visible style.
   Tap targets below WCAG threshold on common actions. iOS Safari
   keyboard-shrink hazards on modals. Screen reader context thin.

None urgent in a "production is broken" sense. All compound across
users + every new slice adds friction on top of the same weak base.

## CRITICAL

### CU1 · Landing page doesn't explain what SWUTrade IS

**Where:** `src/App.tsx:432-610` (default trade view). Anonymous
user hits `/` and sees two empty panels labeled "Offering" and
"Receiving" with `$0.00` totals.

**Walkthrough:** First-timer arrives. They see two panels, a dark
space UI, a sign-in button, no narrative. "Is this a shop? A deck
builder? Why two sides?" "Offering" reads as currency (retail
framing) before it reads as "cards I'll give in a trade." The
AddCardsTile buttons say "Add cards to Offering" — the same opacity.

**Fix:** On empty state, add a one-line subtitle under the
header or as an explicit empty-state card:
"Compare what you'd trade: your cards on the left, their cards
on the right." Could go inside the empty panel where the
AddCardsTile lives now. Don't onboard with a modal; keep it
ambient.

**Effort:** 1 hour.

### CU2 · Sign-in motivation reads generic

**Where:** `src/components/AccountMenu.tsx:48-50`. The sign-in
popover shows "Sync your lists across devices, share a profile
page, and match trades with other users."

**Walkthrough:** Nothing in this copy ties the sign-in to the
cards the user is currently looking at. Trades are the main use
case for signed-in users, but the CTA talks about "syncing" —
which reads as account management, not value.

**Fix:** Reframe the popover. Something like "Find trading
partners and propose trades directly from a card you want. Plus
your lists sync across devices." Lead with community/trade, lead
"sync" (basic-table-stakes) with trail. Mention it unlocks
Community + trade proposals.

**Effort:** 15 minutes (copy change).

### CU3 · Three-axis consent bundling is invisible, user may over-share

**Where:** `src/components/SettingsView.tsx:351-402` (EnrollableGuildCard).
Toggling `enrolled: true` auto-enables `includeInRollups` and
`appearInQueries`. User sees two toggles they didn't flip.

**Walkthrough:** User in Settings. Checks "Enrolled" for their
test server. The card expands, revealing two more toggles,
pre-checked. They didn't enable them explicitly. If they uncheck
one, it turns off that dimension but the bundle decouples silently.
Users running through signup quickly can end up with
`appearInQueries=true` — meaning their handle shows in
community-member directories — without realizing it.

**Fix:** Two changes:
1. Add a 1-line note when the sub-toggles auto-enable:
   "Enrolling turns these on by default. You can customize below."
2. Use visually distinct styling for "auto-enabled" vs. "manually
   toggled" so users see the bundle relationship. A small
   "auto" badge next to sub-toggles the first time.

**Effort:** 1 hour.

### CU4 · CounterBar drops users cold into a composer

**Where:** `src/components/CounterBar.tsx:40-57`. User clicks
"Counter" in a Discord DM, gets an ephemeral link, opens it in
their browser, lands at `/?counter=<id>` with cards pre-filled via
a side-flip from the original.

**Walkthrough:** User opens the link on mobile. Three-word header:
"Countering @alice's proposal". Below it, the trade composer with
cards they've never seen, on sides that are flipped from what the
DM showed them. "Wait, is this what Alice proposed? Or is this
what I'm about to counter with? Why are these cards here?" Without
a contextual hint, they either (a) send a confused counter or (b)
abandon.

**Fix:** Add a banner above or under the CounterBar when
`original` loads: "Responding to @alice's proposal. Sides are
flipped — you're now offering what they asked for. [View original]"
with a link to `/?trade=<originalId>`. Dismissible with localStorage
remember-me so repeat counterers don't see it.

**Effort:** 2 hours.

### CU5 · No global `:focus-visible` styling on dark palette

**Where:** `src/index.css`. One focus rule (line 153), a
`:focus-within` for hover-reveals. Nothing else.

**Walkthrough:** Keyboard-only user tabs through the app. Default
browser outline is ~1px gray, invisible on space-900 backgrounds.
They can't tell which element is focused. Screen reader users can
follow along; sighted keyboard users cannot.

**Fix:** Add a global rule:
```css
*:focus-visible {
  outline: 2px solid rgb(245 158 11 / 0.9); /* gold */
  outline-offset: 2px;
  border-radius: 4px;
}
```
Test on all clickable surfaces. Opt-out via `.no-focus-ring` for
the rare case where the outline clashes (e.g., gold buttons that
already have gold borders).

**Effort:** 1 hour including the sweep.

### CU6 · Tap targets under 44×44 on frequent actions

**Where:** `src/components/ListRows.tsx:14,23,37,121,196` — qty
stepper buttons, remove buttons, and priority-star buttons are all
`w-6 h-6` (24px). `src/components/AccountMenu.tsx:38` — signed-out
button is `h-8` (32px).

**Walkthrough:** Mobile user tapping "-1" on a qty stepper with
their thumb misses ~1 in 10 taps on a narrow viewport. No
forgiving hit zone. WCAG 2.5.5 requires 44×44 minimum for mobile.

**Fix:** Keep the visual button size if it's part of the design
language (chip-like), but add an invisible hit area:
```tsx
<button className="w-6 h-6 relative before:absolute before:inset-[-10px] before:content-['']">
```
This bumps the hit zone to 44+px without changing the visual.

**Effort:** 1-2 hours across qty/remove/star buttons.

### CU7 · "Counter" button label in Discord is ambiguous

**Where:** `lib/proposalMessages.ts:103`. Button label is just
"Counter".

**Walkthrough:** Recipient sees three buttons: green Accept, grey
Counter, red Decline. "Counter" in negotiation/finance = "accept
this alternative." In our app = "open a composer where I edit
freely." A user hesitates: "If I click this, am I committing to
a pre-computed counter? Or editing?"

**Fix:** Change label to "Counter offer" or "Make counter-offer".
Discord button labels support a few more characters. Alternative:
use an emoji prefix, e.g., "↩ Counter".

**Effort:** 5 minutes.

## CORE FRICTION

### CF1 · "Wants/Available" vs "Receiving/Offering" concept mapping isn't explicit

**Where:** Throughout. The ListsDrawer + nav + hooks call them
"wants" (what you seek) and "available" (what you have). The trade
balancer calls them "Receiving" (blue) and "Offering" (emerald).
Mapping: wants → Receiving, available → Offering. The colors match
(wants uses blue in ListsDrawer; Receiving is blue), but the names
aren't the same.

**Walkthrough:** User curates their "wants list" in the drawer.
Flips back to the trade balancer. Sees "Receiving" on the blue
side. They have to infer the mapping. Most users will, eventually.
Some will double-add cards to the wrong side.

**Fix:** Secondary labels or shared glossary tooltip. Either:
- Keep the naming but add a subtitle on each panel: "Offering
  (your available cards)" / "Receiving (your wants)"
- Or unify the naming in a future design pass — pick one pair.

**Effort:** 30 minutes if copy-only.

### CF2 · Seven files repeat the same header chrome

**Where:** `App.tsx`, `ProfileView`, `CommunityView`,
`SettingsView`, `TradesHistoryView`, `TradeDetailView`, `ListView`.
Each has a near-identical block: Logo + SWUTrade wordmark + Beta
badge + a back or action button on the right.

**Walkthrough:** Designer or dev wants to tweak the header
(e.g., add a notification icon). They edit 7 files, miss one,
it drifts. Visual consistency depends on memory.

**Fix:** Extract `<PageHeader>` with slots for title kicker + right
actions. Replace all 7 usages. Trivial extraction; saves every
future change from being a sweep.

**Effort:** 2-3 hours.

### CF3 · No shared Button component — heights drift h-7/h-8/h-9

**Where:** `AccountMenu` (h-8), `ListView` (h-9 CTA),
`SettingsView` (h-7 and h-8 mixed), `CommunityView` (h-8 back, h-7
sort pills), `MigrationDialog` (py-2.5 ≈ h-10), `ProposeBar` inline
(py-1.5), kebab (implicit).

**Walkthrough:** Side-by-side views look like cousins, not
siblings. Primary CTA in ListView is h-9 (standard 36px); the same
semantic role in SettingsView may be h-8 (32px). Visual hierarchy
doesn't read consistently.

**Fix:** `<Button variant="primary|secondary|danger" size="md|sm">`
with locked heights (md=h-9, sm=h-8, xs=h-7). Audit + migrate.
Pairs well with CF2.

**Effort:** 4 hours (extract + audit + sweep).

### CF4 · StatusBadge duplicated in TradeDetailView and TradesHistoryView

**Where:** `TradeDetailView.tsx:195+` (StatusBadge) and
`TradesHistoryView.tsx:146+` (StatusChip). Same enum
(pending/accepted/declined/cancelled/expired/countered), different
component, slightly different styling.

**Walkthrough:** User sees the status chip in history, clicks
through to detail. The same state has different visual weight.
Micro-inconsistency that tells them "these aren't the same thing"
even though they are.

**Fix:** Single `<StatusBadge status size="md|sm">` component.
Both views use it.

**Effort:** 1 hour.

### CF5 · "Add a note" disclosure is easy to miss

**Where:** `ProposeBar.tsx:311-356`, `CounterBar.tsx:297-319`. The
disclosure toggle is a small `text-gray-400` button below the main
bar, sharing space with error/trade-id footers.

**Walkthrough:** User composing a proposal thinks "I should
explain the context." Looks for a message field. Doesn't see
one. Sends anyway. Note never reaches recipient.

**Fix:** Make the disclosure visibly gold (text-gold/70) and
slightly left-indented so it reads as an action, not chrome. Or
put it inline in the main bar as a pen icon next to Send.

**Effort:** 30 minutes.

### CF6 · "Back to community" link after send is contextually wrong

**Where:** `ProposeBar.tsx:192-196`, `CounterBar.tsx:234-239`. On
successful send, the bar shows a "Back to community" link.

**Walkthrough:** User just sent a proposal. Their likely next
action: check trades history for the response, propose another
trade to another user, or close the tab. "Back to community" is
the least likely next step — they came FROM the community (or
from a profile). Default should route to `/?trades=1`.

**Fix:** Change primary to "View your trades" →
`/?trades=1`. Keep "Back to community" as a secondary link if
useful.

**Effort:** 15 minutes.

### CF7 · Discord embed text is written in third person, recipient must translate

**Where:** `lib/proposalMessages.ts:75-84`. Fields are "They're
offering" / "They're asking for".

**Walkthrough:** Recipient opens the DM. Reads "Trade proposal from
@alice. They're offering: …". They have to mentally map "they" =
"@alice" = "the other side" and then translate to "what I'd
receive." Works, but requires a beat.

**Fix:** Rewrite in second person from the recipient's view:
"You would receive ($X)" / "You would give ($Y)". Immediately
clear; buttons make sense right away.

**Effort:** 15 minutes.

### CF8 · Loading/empty/error state visual language is ad-hoc

**Where:** Varies per view. TradeDetailView: `animate-pulse` text.
TradesHistoryView: bordered box + pulse. SettingsView: `<LoadingLine>`
inline function. CommunityView: animate-pulse text. No shared
skeleton, no shared empty-state card.

**Walkthrough:** Each view feels like a slightly different product
when it's loading or empty. Users can't easily tell "is this
loading or is this empty?"

**Fix:** Three shared primitives: `<LoadingState>`, `<EmptyState
icon label cta>`, `<ErrorState>`. Each view opts into them instead
of hand-rolling.

**Effort:** 3 hours.

### CF9 · Pricing mode + percentage controls sit at the bottom, far from where users work

**Where:** `TradeBalance.tsx:172-198` — pricing controls live
inside the balance bar at the bottom of the app.

**Walkthrough:** User on mobile building a trade. They add cards
to Offering (scrolls down inside the panel). They want to try
"Low" vs "Market" pricing. They have to scroll past both full
panels to reach the bottom, toggle, then scroll back up to see
card prices update. Each pricing fiddle = full scroll dance.

**Fix:** Move the pricing pill to the header area next to the
account menu, or add a sticky floating affordance. Discussion
whether it belongs in `TradeBalance` (where it semantically fits)
or should be promoted to top-level chrome. I'd promote it.

**Effort:** 2-3 hours.

### CF10 · Clear All's two-click confirm is surprising, no visible countdown

**Where:** `src/components/ClearAllButton.tsx:13-68`. First click
arms the button ("Tap to confirm"); 3-second window; second click
clears. No visible timer.

**Walkthrough:** User taps "Clear All" to dismiss a test trade.
Button turns red, text changes. They look for a confirmation
dialog. Don't see one. Walk away. 3 seconds later, they tap
Clear All again to try a "real" clear. Cards vanish. Surprise.

**Fix:** Either (a) add a visible countdown ring/bar on the armed
state, or (b) use a proper confirm dialog (one-click destructive
with confirm). Dialogs are more conventional.

**Effort:** 1 hour for dialog replacement; 30 min for countdown ring.

### CP4 · Large proposals truncate in Discord embeds silently

**Where:** `lib/proposalMessages.ts:39-49`. `formatCardList`

**Fix:** Check length in formatCardList; if the rendered string

## Suggested order

If I were picking a subset to ship, by ROI:

1. **CU5** (global `:focus-visible`) — 1 hour, covers every view
2. **CU6** (44×44 hit zones on qty/remove) — 2 hours, mobile + a11y win
3. **CF2 + CF3 + CF4** (PageHeader, Button, StatusBadge extraction)
   — 6 hours bundled, foundational primitives that every future
   slice benefits from
4. **CF7** (first-person Discord embed text) — 15 min
5. **CU7** (Counter button label) — 5 min
6. **CU4** (CounterBar context banner) — 2 hours, actually
   communicates what counter flow is about
7. **CU3** (consent bundling explainer) — 1 hour, closes a real
   over-sharing risk
8. **CU1** (landing page empty-state context) — 1 hour
9. **CU2** (sign-in copy) — 15 min

That's ~12-13 hours for the highest-impact UX pass. Everything
else is a natural fit for opportunistic inclusion when the
relevant feature gets touched.

## Cross-reference with CODE_REVIEW_2026_04_17.md

Several findings overlap between the two reviews. Pairing them
saves effort:

- **CF3 (Button component) + code review A9 (API client)**: both
  ask for shared primitives. Different axes (UI vs data), but
  same theme — extract before the 4th slice that needs them.
- **CF5 (Add a note disclosure hidden) + code review C5 (useCallback
  deps audit)**: the note disclosure has the bug we already fixed
  (stale closure on `message`). Formalizing an eslint rule catches
  a recurrence whether it's hidden or visible.
- **CU4 (CounterBar context banner) + code review A2 (ProposeBar +
  CounterBar duplication)**: both flag the two bars as tightly
  related. Extracting a shared `useComposerBar` hook makes adding
  the banner a one-place change.
- **CP4 (embed truncation) + code review P1 (Discord error
  classification)**: both are Discord-message reliability. Bundle
  into a "harden the Discord outbound path" slice.

If you want one "foundational" slice to tackle, it's the design-
system extraction (PageHeader + Button + StatusBadge +
LoadingState/EmptyState/ErrorState). That plus the `:focus-visible`
global + the hit-area fix is ~10-12 hours and every future view
gets cleaner.
